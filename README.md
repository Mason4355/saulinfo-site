# SaulInfo Site

Финальная установка всего стека выполняется из репозитория `shop-update`: `docs/FINAL_DEPLOY.md`.
Для отдельного запуска сайта путь к данным панели можно переопределить через `SHOP_UPDATE_DATA_DIR`.

Отдельный пользовательский сайт для `www.saulinfo.ru`.

Цели проекта:

- отдельный репозиторий и контейнер
- отдельный публичный домен
- интеграция с `shop-update` через аккуратный слой доступа к данным
- без вмешательства в текущую внутреннюю админскую систему

## Что внутри

- Flask-приложение для публичного сайта
- gateway-слой для чтения пользовательских данных из БД `shop-update`
- локальная e-mail авторизация
- страницы кабинета пользователя
- GitHub Actions для автодеплоя на сервер

## Локальный запуск

```bash
cp .env.example .env
docker compose up --build
```

Открыть:

```text
http://localhost:8080
```

## Интеграция с shop-update

Сайт читает данные из SQLite-базы текущей системы через `SHOP_UPDATE_DB_PATH`.

По умолчанию в контейнер монтируется:

```text
/root/shop-update/data -> /integrations/shop-update
```

Это позволяет новому сайту работать поверх тех же пользователей, ключей, баланса и тикетов, не изменяя код `shop-update`.

## Деплой

После настройки GitHub Secrets деплой делается автоматически при push в `main`.

## Обновление одной командой на сервере

После первого клона на сервере обновление сайта делается так:

```bash
cd /root/saulinfo-site
chmod +x ./update ./deploy/update.sh
./update
```

Что делает команда:

- подтягивает свежий `main`
- пересобирает контейнер
- перезапускает сервис
- ждёт `healthcheck`
- завершает обновление только если контейнер поднялся нормально

Если хотите ещё короче, на сервере можно один раз сделать алиас:

```bash
ln -sf /root/saulinfo-site/update /usr/local/bin/update-site
```

Тогда дальше обновление будет одной командой:

```bash
update-site
```
